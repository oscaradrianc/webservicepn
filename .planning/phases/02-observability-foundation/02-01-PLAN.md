<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| appsettings.json → Serilog sink config | La ruta del archivo de log y el outputTemplate vienen de config — si alguien puede escribir appsettings.json, puede redirigir logs |
| Log fields → structured output | Los valores de campos estructurados (Recipients, Subject) pasan por el template de Serilog antes de salir al sink |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01-01 | Information Disclosure | Log.Fatal catch en app.Run() | accept | El catch loguea la excepción de startup — puede contener connection string si Oracle falla. Aceptable: el log file requiere acceso al servidor. No loguear ctx.Configuration en el catch. |
| T-02-01-02 | Tampering | appsettings.json Serilog path config | accept | La ruta `Logs/log.txt` es un valor en appsettings.json. Path traversal no aplica — la ruta se pasa directamente a Serilog.Sinks.File que la valida. No es user-controlled input. |
| T-02-01-03 | Denial of Service | Rolling file sink | accept | Si el disco se llena, Serilog falla silenciosamente (no lanza excepción en el request path). Aceptable para ASVS L1. Monitoreo de disco es responsabilidad de infra. |
</threat_model>
