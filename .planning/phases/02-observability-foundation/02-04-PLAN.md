<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| General.cs Log.Error → Serilog static logger | El Log.Error estático usa el Log.Logger global configurado en Program.cs — si se llama antes de que el host inicialice, usa SilentLogger (no-op, no crash) |
| Log fields Recipients/Subject → log file | Las direcciones de email de destinatarios quedan registradas en el log file con cada fallo de SMTP |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04-01 | Information Disclosure | Log.Error loguea listaCorreos (email addresses) | accept | Las direcciones de email son datos de negocio que DEBEN aparecer en los logs de error para diagnóstico SMTP. ASVS L1 no prohíbe email addresses en logs internos — solo passwords, tokens, y PII crítico. El log file es local al servidor. |
| T-02-04-02 | Information Disclosure | Log.Error loguea asunto del email | accept | El subject puede contener referencias a solicitudes (ej. "Invitación a cotizar SOL-2024-001"). Aceptable en logs internos de error. |
| T-02-04-03 | Elevation of Privilege | OracleMonitor sin guard en producción | mitigate | **Este plan cierra esta amenaza**: el guard `if (app.Environment.IsDevelopment())` asegura que OracleMonitor.IsActive = true solo se ejecuta en Development. Sin el guard, el diagnóstico Devart podría exponer queries SQL en output de producción. |
| T-02-04-04 | Information Disclosure | Stack trace en Log.Error(ex, ...) | accept | El primer parámetro `ex` (Exception) hace que Serilog serialice el stack trace completo. Va al log file (server-side), no al cliente. ASVS L1 requiere que los stack traces no lleguen al cliente — este log es interno. |
</threat_model>
